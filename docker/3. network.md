# docker network

## docker network 필요성
도커 내부 컨테이너끼리 통신하기 위해 컨테이너별로 ip를 주고, 서로 통신할 수 있는 네트워크를 만들어줌

```bash
                Docker Network

      nginx  <------>  spring  <------> postgres
  (172.17.0.2)      (172.17.0.3)      (172.17.0.3)
```
host의 네트워크와 docker 내부의 네트워크는 별도의 네트워크 
docker 컨테이너에서 외부로 나가기 위해서는 게이트웨이를 통해야함.
```bash
            AWS VPC                              host network
        (172.31.0.0/16) -> ec2를 위한 큰 네트워크
                │
        Internet Gateway
                │
                ▼
           EC2 (ens5)
        (172.31.44.71) -> aws vpc와 연결된 실제 ec2 네트워크 인터페이스
                │
                │ Docker Engine
                ▼
  --------------------------------------------------------------
                                                  docker network
            docker0 -> 가상 네트워크 스위치(bridge)
         (172.17.0.1) -> 컨테이너들이 사용하는 게이트웨이 역할
               │         컨테이너들을 위한 작은 네트워크
               │
        ┌──────┴───────┐
        │              │
        ▼              ▼
    172.17.0.2   172.17.0.3
      nginx        spring
```
잠깐!!
gateway는 밖으로 나갈 때 반드시 거쳐야 하는 출입문. 현관문 같은거!
```bash
방 (container)
    ↓
현관 (gateway)
    ↓
복도 (host)
    ↓
밖 (internet)
```

## docker network
docker를 설치하면 기본적으로 3개의 네트워크를 만들어줌
1. bridge
2. host
3. none

### 1. docker bridge network
- docker의 기본 네트워크. 가상 스위치. 
- docker 내부 네트워크기 때문에 외부 pc로는 접속할 수 없음. host를 통해서 접근해야함.
- docker run 같은 명령어 실행 시 bridge로 연결
  명령어 실행 시 --network 옵션을 적지 않으면 기본적으로 bridge 네트퉈크에 연결함

```bash
# 격리된 네트워크를 만들고, 그 안에서 서버 혹은 컨테이너끼리 통신한다는 개념이 docker bridge, vpc 끼리 동일
docker 버전
           docker0 -> 가상 스위치 역할
        ┌─────┼─────┐
        │     │     │
     nginx  spring postgres

aws 버전
             vpc 
        ┌─────┼─────┐
        │     │     │
     ec2 1  ec2 2 ec2 3

```

### 2. docker host network
- bridge를 쓰지 않고 컨테이너가 host의 네트워크를 그대로 사용
  즉 host의 ip를 컨테이너 ip로 사용. 중간에 docker0 없음.
- 네트워크 성능이 좋지만, 격리가 사라지는 단점이 생김
  따라서 성능이 매우 중요한 특수한 경우만 사용

### 3. docker none network
- 말그대로 네트워크가 없음.
- 파일 변환, 암호화, 영상 인코딩 등 네트워크가 필요없는 작업을 하는 컨테이너에서 사용. 
- 보안상 안전
```bash
    Container
        ↓
    IP 없음
        ↓
    인터넷 안 됨
        ↓
다른 컨테이너와 통신 안 됨
```

### 4. user-defined bridge (실무에서 많이 사용!)
ip가 아닌 dns 이름으로 호출하기 위해서 사용자 지정 네트워크 생성한 뒤 호출한다.
- 실무에서는 ip로 호출하지 않고 dns 이름으로 호출함.
- 컨테이너가 다시 실행되면 ip는 계속 변경되기 때문

4-3. 컨테이너끼리 ip 없이 호출해보기
같은 네트워크를 사용해서 실행한 컨테이너 중 alpine 컨테이너에서 ip 없이 nginx 호출하면 docker 내부 dns가 동작하여 web -> nginx ip를 찾아서 호출함.
컨테이너끼리 호출할 때 ip를 사용하지 않고 모두 이름으로 통신함!!
```bash
# 명령어
wget -O- http://web

# 호출 flow
   web
    ↓
 DNS 조회
    ↓
172.18.0.2
    ↓
 HTTP 요청
```

4.4 docker dns 확인

```bash
# 결과

(컨테이너 ip) (컨테이너 id)

# 3. ocker DNS가 컨테이너 이름을 IP로 변환하는지 확인
getent hosts web
# 결과
172.18.0.2      web
```

# docker network 실습
## container 끼리 통신해보기

### 1. 테스트용 컨테이너 생성
가벼운 alpine을 생성하기
- d : 백그라운드에서 실행하기
- i : 표준 입력 유지하기
- t : 터미널 실행하기
- sh : alpine은 bash 대신 sh
```bash
# 컨테이너 실행
docker run -dit --name test alpine sh

# 실행 확인
docker ps
```

### 2. 신규 컨테이너에서 기존 nginx에 ping 보내기
- ping : ICMP 프로토콜을 이용해 상대가 살아있는지 확인하는것.
         http 되는거랑은 별개
         ping이 되면 네트워크 연결은 되어 있다는 뜻
- curl, wget : http 프로토콜로 웹 요청
```bash
# test 컨테이너로 접속하기
docker exec -it test sh

# ping으로 네트워크 연결 확인
ping 172.17.0.2(nginx ip)

# wget으로 http 통신 확인
wget -O- http://172.17.0.2
```


### dns명으로 네트워크 호출이 안될 시 확인할 순서!

① docker ps
→ 컨테이너가 실행 중인가?

② docker inspect web(dns name)
→ IP가 무엇인가?

③ getent hosts web(dns name)
→ DNS가 정상인가?

④ ping web(dns name)
→ 네트워크 연결이 되는가?

⑤ wget http://web(dns name)
→ HTTP 서비스가 되는가?

### docker network 명령어

### 네트워크 명령어
| 명령어                                 | 무엇을 확인하는가?                            | 언제 사용하는가?                   |
| ----------------------------------- | ------------------------------------- | --------------------------- |
| `docker network ls`                 | Docker에 존재하는 네트워크 목록                  | 어떤 네트워크가 있는지 확인             |
| `docker network inspect my-network` | 네트워크의 상세 정보(IP 대역, Gateway, 연결된 컨테이너) | 네트워크 구성을 확인할 때              |
| `docker inspect web`                | 컨테이너의 상세 정보(IP, DNS, Mount, Port 등)   | 특정 컨테이너의 설정 확인              |
| `ping <IP 또는 이름>`                   | ICMP 통신이 되는지 확인                       | 네트워크 연결 확인(서비스와는 무관)        |
| `wget -O- http://web`               | HTTP 요청을 보내 웹 서버가 응답하는지 확인            | 웹 서버(Nginx, Apache 등) 동작 확인 |
| `hostname`                          | 현재 컨테이너 자신의 이름(HostName) 확인           | 지금 어느 컨테이너 안에 있는지 확인        |
| `cat /etc/hosts`                    | 로컬 hosts 파일 확인                        | 자기 자신의 이름 매핑 확인             |
| `cat /etc/resolv.conf`              | 어떤 DNS 서버를 사용할지 확인                    | DNS 설정 확인                   |
| `getent hosts web`                  | DNS를 이용해 이름을 IP로 변환                   | DNS가 정상 동작하는지 확인            |
