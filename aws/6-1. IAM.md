# IAM

## IAM(Identity and Access Management) 이란?
AWS 리소스에 누가(Identity), 무엇을(Access), 어떻게 접근할 수 있는지 관리하는 서비스
```bash
Identity        Access      Management
   신원            접근           관리
-> 누구야??    -> 권한 있음?  -> 권한 관리는?
```

## 필요한 이유?
aws에 ec2, s3, rds... 많은 리소스가 있는데 아무나 이걸 수정하거나 삭제할 수 있으면 안됨
개발자랑 운영자의 권한을 나눠서 줘야함.
이걸 관리하는게 IAM

## 핵심 개념
policy를 user나 role에 연결
```bash
                 Policy
                   |
        -----------------------
        |                     |
       User                  Role
        |
      Group
```
### 1. policy(정책)
! IAM에서 가장 중요한 개념 ! "권한 그 자체"
어떤 행동(Action)을 어떤 대상(Resource)에 허용할 것인지 정의한 문서
문서는 json으로 구성되어 있음.
```bash
{
  "Effect": "Allow",
  "Action": [
    "ec2:DescribeInstances"
  ],
  "Resource": "*"
}
```
| 항목        | 의미          |
| --------- | ----------- |
| Effect    | 허용/거부       |
| Action    | 무엇을 할 수 있는지 |
| Resource  | 어떤 대상인지     |
| Condition | 조건          |

### 2. user(사용자)
사람 혹은 application을 뜻함.
- aws console 로그인할 때 사용
- aws cli 사용할 때 사용
- api 호출할 때 사용

### 3. group(그룹)
user를 묶어서 관리하는 것.
user를 group으로 묶은 다음 group에다가 권한을 붙이면 한명한명에게 권한을 따로 줄필요가 없음!
그룹에만 주면 그룹 안 유저들은 전부 그 권한을 가지게 됨.

### 3. role(역할)
대상이 사람이 아니라 AWS 서비스나 외부 주체에게 임시 권한을 부여하는 방식
user는 고정, role은 임시! 필요할 때만 권한을 획득하는 임시성.
- ec2가 s3 접근할 때 사용
- eks pod가 aws api 호출할 때 사용

### root user, IAM user, IAM role 차이점
user : 개발자가 aws console에 접속하거나 cli 사용
role : aws ec2가 s3, ssm 들에 접근하려고 할때 사용
| 구분         | Root User      | IAM User   | IAM Role      |
| ---------- | -------------- | ---------- | ------------- |
| 대상         | AWS 계정 자체      | 사람         | 서비스/사람/외부 시스템 |
| 생성 시점      | AWS 가입 시 자동 생성 | 관리자가 생성    | 관리자가 생성       |
| 로그인        | 가능             | 가능         | 불가능           |
| 비밀번호       | 있음             | 있음(선택)     | 없음            |
| Access Key | 가능             | 가능         | 직접 사용 X       |
| 권한         | 전체 권한          | Policy로 제한 | Policy로 제한    |
| 주 용도       | 계정 관리          | 사람의 AWS 사용 | AWS 서비스 권한 부여 |


## policy 종류
- AWS Managed Policy
- Customer Managed Policy
- Inline Policy

### 1. AWS Managed Policy (AWS 관리형 정책)
aws가 자주 사용하는 권한 조합을 미리 만들어서 제공하는 정책

- 장점 : 빠르고 편하며, policy에 권한이 변경됐을 시 aws에서 자동으로 알아서 업데이트됨 신경쓸 필요 없음.
- 단점 : 특정 이름의 리소스에만 적용해줘! 같은.. 세밀한 제어 어려움.

### 2. Customer Managed Policy (고객 관리형 정책)
aws가 아닌 사용자가 직접 만드는 정책
실무에서 사용 많이함.

- 장점 : 특정 이름의 리소스에만 적용해줘! 같은.. 세밀한 제어 가능, 재사용 가능

### 3. Inline Policy (인라인 정책)
특정 User / Role / Group에 직접 붙이는 일회성 정책
ooo policy 같이 이름이 있지만 독립적인 policy 객체가 아님.

- 단점 : 재사용 불가

| 구분     | AWS Managed                  | Customer Managed | Inline          |
| ------ | ---------------------------- | ---------------- | --------------- |
| 누가 생성? | AWS                          | 사용자              | 사용자             |
| 재사용    | 가능                           | 가능               | 불가능             |
| 수정     | AWS가 관리                      | 사용자가 관리          | 붙은 대상에서만 수정     |
| 세밀한 제어 | 낮음                           | 높음               | 높음              |
| 실무 사용  | 많음                           | 매우 많음            | 특정 상황           |
| 대표 예   | AmazonSSMManagedInstanceCore | 회사 전용 S3 정책      | 특정 Lambda 전용 권한 |


## IAM role 좀 더 이해하기!
role은 2개의 policy가 같이 있어야함.
- Trust Policy (신뢰 정책) : 누가 사용하는지 결정
- Permission Policy (권한 정책) : 무엇을 사용???

```bash
              IAM Role

        ┌───────────────────┐
        │                   │
        │  Trust Policy     │  ← 누가 이 Role을 사용할 수 있는가?
        │                   │
        ├───────────────────┤
        │                   │
        │ Permission Policy │  ← Role을 사용하면 무엇을 할 수 있는가?
        │                   │
        └───────────────────┘
```

### Trust Policy (신뢰 정책)
어떠한 role에 대해서 누가 사용 가능한지 결정하는 정책
```bash
{
  "Effect": "Allow",
  "Principal": {
    "Service": "ec2.amazonaws.com"  -> 누가?? ec2가
  },
  "Action": "sts:AssumeRole" -> role assume하는 것을 허용
}
```
### Permission Policy (권한 정책)
무엇을 할 수 있는 권한인지 정해주는 정책
```bash
{
  "Effect": "Allow",
  "Action": [
    "s3:GetObject"  -> s3 객체를 읽을 수 있다.
  ],
  "Resource": [
    "arn:aws:s3:::my-bucket/*" -> my-bucket 하위에 있는 리소스에 대해서
  ]
}
```

## 실습!!!
ec2가 ccess Key 없이 IAM role을 통해 S3에 접근하게 하기

```bash
EC2 Instance
   ↓
IAM Role
(Trust Policy: EC2 허용)
   ↓
Permission Policy
(S3 조회 권한)
   ↓
S3 Bucket 조회
```

### 1. S3 만들기
S3 만든 다음 텍스트파일 업로드하기
```bash
버킷 네임스페이스 : 계정 리전 네임스페이스
객체 소유권 : acl 비활성화
   -> 객체별로 권한을 주는 방식으로 예전 방식. 요즘은 policy로 관리하기 때문에 비활성화
퍼블릭 엑세스 차단 설정 : 모든 버플릭 엑세스 차단
버킷 버전 관리 : 활성화
   -> 파일 변경 이력 저장하는 기능. 중요 데이터나 로그, 테라폼 state 같은 곳에 많이 사용 
암호화 유형 : amazone s3관리형 키(sse-s3)
   -> s3에 저장된 데이터가 aws 내부에서 암호화된 상태로 보관되는데 어떤 방식을 사용할 것인지 선택
   -> sse-s3 : 가장 기본적인 방식으로 일반 파일, 로그, 테스트 환경에서 주로 사용
   -> sse-kms : kms를 이용한 방식으로 금옹, 개인정보 같은 중요한 데이터 저장할 때 사용
   -> dsse-kms : Dual-layer Server-Side Encryption로 이중 암호화로 암호화가 아주아주 좋을 때 사용
버킷 키 : 활성화 (SSE-S3를 사용할 것이기 때문에 사실 영향 없음)
   -> kms 암호화 비용과 api 호출을 줄이기 위한 기능
```

### 2. IAM role 생성하기
ec2가 s3를 읽을 수 있는 role 생성하기
```bash
신뢰할 수 있는 엔터티 유형 : aws 서비스
  -> 이 부분이 Trust Policy (신뢰 정책)
사용 사례 : ec2
권한 정책 : AmazonS3ReadOnlyAccess
```

### 3. ec2에 role 연결
ec2 선택 -> 작업 -> 보안 -> IAM 역할 수정 -> 방금 만든 role 추가
