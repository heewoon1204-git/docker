# Subnet

## subnet 이란?
- vpc :  대단지 아파트 단지
- subnet : 101동/102동/103동 같은 아파트 동!
           구역을 나눠서 관리하는 하나의 아파트 동이라고 생각하면 될듯.
- ec2 : 각각의 집. 101호, 102호..
- route table : 네비게이션. subnet에 설정
- internet gateway : 아파트 정문. vpc에 연결
- internet : 도로

장애를 대비하기 위해 여러 az에 리소스를 분산 배치하는 것이 보통.
```bash
VPC
10.0.0.0/16
├── Public Subnet A
│      10.0.1.0/24
│
├── Public Subnet C
│      10.0.2.0/24
│
├── Private Subnet A
│      10.0.11.0/24
│
└── Private Subnet C
       10.0.12.0/24
```

## public subnet 
인터넷과 직접 통신할 수 있는 서브넷.
-> internet gateway로 가는 route가 있는 subnet
public subnet이 되기 위해서는 세가지 조건이 필요함.

1. internet gateway 연결
2. route table에 추가
3. ec2에 public ip 부여

### 1. internet gateway 생성 및 연결
인터넷 게이트웨이 : vpc와 인터넷의 경계에 있는 장비, vpc에 연결
igw는 공유기 같은 느낌. 공유기를 사도 선을 연결하지 않으면 인터넷을 사용하지 못하는 것처럼 vpc에 연결해야함.
```bash
      인터넷
        │
        │
[ Internet Gateway ]
        │
        │
     VPC 내부
```

### 2. route table생성 및 igw 라우팅 추가
ec2가 인터넷으로 나가려고 할 때 igw를 자동으로 찾아서 나가지 않음.
인터넷으로 나가는 길을 모르는데 이때 네비게이션 역할을 해주는게 route table.

route table을 만들고 나면, subnet에 연결해야함. 
vpc 안애는 여러 subnet이 존재할 수 있고, subnet마다 다른 라우팅 정책을 적용해야되기 때문에 vpc가 아니라 subnet에 연결!
```bash
Internet
      │
Internet Gateway
      │
      ▼
-----------------------------
VPC (10.0.0.0/16)
│ 
├── Public Subnet
│       │
│       └── Public Route Table
│              │
│              └── Internet Gateway
│
└── Private Subnet
        │
        └── Private Route Table
               │
               └── NAT Gateway
```

route table을 만들면 기본적으로 vpc 내부 통신 할 수 있게 local이 설정됨
(사용자가 만드는게 아니라 aws에서 자동으로 생성하는 것으로 삭제 및 수정 불가)
-> local이 있어야 vpc 내부 ec2끼리 통신이 가능함

인터넷 게이트웨이는 인터넷으로 나가려면 저 게이트를 통과해야된다고 알려주는 것.
| Destination   | Target           | 의미            |
| ------------- | ---------------- | ------------- |
| `10.0.0.0/16` | local            | **VPC 내부 통신** |
| `0.0.0.0/0`   | Internet Gateway | **외부 인터넷 통신** |


### 3. ec2에 public ip 부여
ec2에 public ip를 부여해야 외부에서 접근이 가능함.

ec2에는 2개의 Ip를 사용
- public ip : 인터넷에서 사용하는 주소. 즉 외부 접근
              ec2가 직접 가지고 있는 ip가 아님.
- private ip : vpc 내부에서 사용하는 주소
               실제 ec2에는 private ip만 있음.


## private subnet 
인터넷에서 직접 접근할 수 없는 서브넷.
internet gateway라는 route가 없기 때문에 인터넷으로 접근 불가.

## 접속 실습
private subnet에 ec2를 생성했고, ec2는 public ip 없이 생성된 상태.

1. 외부에서 ssh로 접근 -> 실패
외부 pc에서 인터넷으로 private ip로 접근 불가능.
애초에 private ip는 vpc 내부에서 사용되는 ip임
vpc 외부에서 이걸로 접속한다??? 당연히 불가능함.
```bash
ssh -i new_docker_key_0714.pem ubuntu@private ip
```


2. public ec2 내부에서 pricate ec2로 ping -> 성공
ec2 보안그룹 inbound 규칙에 ICMP가 뚫려있으면 ping 성공
ICMP 없이 HTTP, SSH만 있으면 ping 불가능.
```bash
ping Private EC2의 Private IP
```
private subnet의 route table을 보면 "10.0.0.0/16 -> local" 설정이 되어 있어서 vpc 내부 통신이 가능하기 때문.
10.0.0.0/16 -> local -> 요거 있으면 vpc 내부는 통신된다고 보면 됨.
단, security group 허락이 있으면..

3. public ec2 내부에서 ssh로 접속 -> 성공
ssh키를 public ec2로 옮긴 다음 진행해야함.
private ec2 보안그룹 inbound ssh에 vpc CIDR이 뚫려있어야 진행 가능.

* 실무에서는 키를 절대 Public에 복사하지 않고 SSH Agent Forwarding이나 aws ssm을 이용함
```bash
# 키 복사
scp -i new_docker_key_0714.pem new_docker_key_0714.pem ubuntu@<Public EC2 IP>:~/

# 권한 변경
chmod 400 new_docker_key_0714.pem

# ssh 접속
ssh -i new_docker_key_0714.pem ubuntu@pricate ip
```


4. private ec2 내부에서 외부로 접속 -> 실패
private ec2가 생성된 Private subnet의 route에는 internet gateway가 없어서 실패
```bash
ping 8.8.8.8
-> Google Public DNS 서버 IP. 인터넷에서 항상 접근이 가능해서 네트워크 테스트로 많이 사용

curl https://checkip.amazonaws.com
-> aws 서버에 접속해서 현재 인터넷에 보이는 public ip확인
```


5. private ec2에서 apt update -> 실패
패키지 서버가 인터넷에 연결되어있기 때문에 apt update 하기 위해서는 인터넷에 연결되어있어야 성공함.
```bash
sudo apt update
```
